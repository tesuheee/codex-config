# 気になることリスト リサーチツール（多重マルチエージェント版）

引数: $ARGUMENTS

## 概要

`■メモ/🧭気になることリスト.md` の項目を選び、調査・分解・具体化して `■リサーチ/` に個別ファイルとして書き出す。

本スキルは以下の二層で並列化する。

- 外側並列: 項目ごとに担当エージェントを分ける
- 内側並列: 各項目の「調べること / 決めること」をサブテーマに分割し、サブテーマごとに複数エージェントを動かす

## パス情報

- 気になることリスト: `C:\Users\nedie\Obsidian\■メモ\🧭気になることリスト.md`
- リサーチ出力先: `C:\Users\nedie\Obsidian\■リサーチ\`
- タスクリスト: `C:\Users\nedie\Obsidian\■メモ\📆タスクリスト.md`
- Vault ルート: `C:\Users\nedie\Obsidian\`

## 実行モード

- 標準モード（既定）: `spawn_agent` / `send_input` / `wait` / `close_agent` を使って多重マルチエージェントで実行する
- 互換モード: マルチエージェント系ツールが使えない場合のみ、同じ手順を単一エージェントで順次実行する

## 並列実行ポリシー

- 外側並列（項目単位）: 同時実行は最大3項目まで
- 内側並列（サブテーマ単位）: 1項目あたり最大3サブテーマ、各サブテーマで最大2エージェント（Web担当 + Vault担当）
- 全体上限: 同時に走らせるエージェント総数は8以内を目安に制御する
- タイムアウト時: 子エージェントは1回だけ再実行し、再失敗時は「未解決の疑問」に明示して先へ進む

## Phase 0: 対象選択

### 引数がある場合

`$ARGUMENTS` のテキストを項目名として扱い、気になることリストから一致する項目を探す。
見つからない場合は `$ARGUMENTS` をそのまま新規テーマとして Phase 1 に進む。

### 引数がない場合

1. `■メモ/🧭気になることリスト.md` を `Read` する
2. `### 未リサーチ` セクションのみをパースし、項目タイトルを抽出する（`→` の前まで）
   - `### リサーチノート作成済み` セクションは選択候補に含めない（ノート作成済みのため）
   - `### 運用メモ` セクションは対象外
3. `未リサーチ` セクションの項目が0件の場合は Phase 1〜3 をスキップし、Phase 4 へ直接進む
4. 項目が1件以上ある場合は `AskUserQuestion` で対象項目を選択する（multiSelect: true）
   - 選択肢: `未リサーチ` セクションの各項目のタイトル
   - 最大4つ。項目が5つ以上ある場合は最初の3つ + 「その他（テキスト入力）」

## Phase 1: エージェント編成（外側）

1. 選択項目ごとに「項目担当エージェント」を起動する（最大3並列）
2. 項目担当エージェントに以下を渡す
   - 対象項目名
   - 対象項目の補足テキスト（あれば）
   - 既存リサーチノートの有無
3. 親エージェントはキューを持ち、空きが出たら次の項目担当を起動する

### 項目担当エージェントへの指示テンプレート

```text
あなたは「項目担当」です。対象は 1 項目のみ。
まず項目を「調べること / 決めること / やること」に分解し、
「調べること / 決めること」から最大3つのサブテーマを作成してください。
以降、各サブテーマで Web担当とVault担当を並列起動し、結果を統合して返してください。
```

---

## Phase 2: 多重調査（内側）

各項目担当エージェントは以下を実行する。

### 2-1. 既存リサーチノートの確認

1. `Glob` で `■リサーチ/*.md` を検索し、対象項目に対応するファイルがあるか確認する
2. ある場合は `Read` して前回の調査状態を把握する
3. 前回からの差分（新しく調べるべきこと）を特定する

### 2-2. 項目の分解

項目のテキスト（タイトル + サブ項目）を以下の3軸に分類する。

- **調べること**: 情報が足りない部分。Web検索やVault検索で解決できるもの
- **決めること**: 選択肢があり、判断が必要な部分。比較情報を集めて判断材料を提示する
- **やること**: 具体的なアクション。手順に落とし込む

### 2-3. サブテーマ別の並列調査

1. 「調べること」と「決めること」からサブテーマを最大3件作る
2. 各サブテーマごとに次の2エージェントを並列起動する
   - Web担当: `WebSearch` / `WebFetch` を使い、出典URL付きで事実を収集
   - Vault担当: `Grep` / `Read` / `Glob` で既存ノートから関連情報を抽出
3. Web担当は1サブテーマあたり検索3回まで
4. 技術系トピックは公式ドキュメントを優先する

### 2-4. サブテーマ統合

項目担当エージェントは子エージェント結果を統合し、重複・矛盾を整理する。
固定カテゴリへの分類は不要。テーマに合った形で自由にまとめる。

### 2-5. 親エージェントへの返却形式

各項目担当は次の構造で返す。

```markdown
# [項目タイトル]

## 疑問点
- 調査後も残った疑問・未確認の論点

## 結論
- 現時点の結論と根拠

## 次にやること
- すぐできること
- 追加調査が必要なこと（未解決の論点も含む）

## 調査メモ
- サブテーマごとの要点（出典はインラインで記載）
- Vault内の関連ノートへの言及
```

## Phase 3: 書き出し（親エージェント）

親エージェントは、項目担当から返却された統合結果を使って書き出す。

### 3-1. リサーチノートの作成

`■リサーチ/[項目の短縮名].md` に結果を書き出す。

ファイル名のルール:

- 項目タイトルから簡潔なファイル名を生成する
- 例: 「Googleカレンダーへの切り替え」→ `Googleカレンダー移行.md`
- 例: 「AI動画編集の現状調査」→ `AI動画編集調査.md`

フォーマット:

```markdown
# [項目タイトル]

元: [[■メモ/🧭気になることリスト]]
調査日: YYYY-MM-DD
- [ ] 対応完了

## 疑問点

- 調査後も残った疑問・未確認の論点（なければ省略可）

## 結論

- 暫定結論と根拠（何の情報を重視したか）

## 次にやること

- [ ] すぐできるタスク
- [ ] 追加調査が必要なタスク（未解決の論点も含む）

## 調査メモ

<!-- テーマに合わせて自由に構成する。比較表・手順・メリデメなど最適な形式を選ぶ -->
<!-- 出典URLはインラインで記載する。独立した「参考」セクションは作らない -->
<!-- Vault内の関連ノートへのリンクも本文中に自然に配置する -->

### [サブテーマ1]
- 調査で得た情報（[出典名](URL)）

### [サブテーマ2]
- 調査で得た情報
```

構成の原則:

- セクションは「疑問点 → 結論 → 次にやること → 調査メモ」の構成とする。疑問点がない場合は省略可
- 「調査メモ」の中身はテーマに応じて自由に構成する（比較表、手順、メリデメ、時系列など）
- 出典URLは調査メモの本文中にインラインで書く。末尾に参考セクションを別途作らない
- Vault内の関連ノートも本文中に自然にリンクする
- 更新履歴セクションは作らない（ファイルの更新日で十分）

既存ファイルがある場合:

- 前回の内容は保持する
- `## 調査メモ` に新しい情報を追記する
- `## 次にやること` を更新する（完了済みがあればチェック）
- `## 疑問点` を更新する（解決済みは削除、新たな疑問は追加）
- `## 結論` を見直して更新する
- 末尾に `### YYYY-MM-DD 追加調査` セクションを追記してもよい

### 3-2. 気になることリストの更新

`■メモ/🧭気になることリスト.md` を `Edit` で更新する。

リストは次の構造を前提に更新する:

```markdown
### 未リサーチ
- [ ] テーマ名

### リサーチノート作成済み（チェック付きは対応完了）
- [ ] テーマ名 → [[■リサーチ/ファイル名]]
- [x] テーマ名 → [[■リサーチ/リサーチ完了/ファイル名]]
```

更新ルール:

- リサーチノートを新規作成した項目は `未リサーチ` から `作成済み` へ移動する
- `作成済み` セクションの項目は必ずリンク付きにする（リンクなしを許容しない）
- `作成済み` の未完了項目は `[[■リサーチ/ファイル名]]` 形式で統一する
- `作成済み` の完了項目（`[x]`）は `[[■リサーチ/リサーチ完了/ファイル名]]` 形式で統一する
- 既存項目でリンクが欠けている場合は、該当ノート名からリンクを補完して保存する
- `未リサーチ` 配下の補足メモ（インデント行）は、項目移動時に保持するか、リサーチノートの「調査メモ」へ転記する
- タイトルの重複は作らない（既存行を更新する）

### 3-3. タスクリストへの次アクション追記（任意）

調査の結果「すぐやれる具体的なタスク」が明確になった場合のみ、`■メモ/📆タスクリスト.md` に追記する。

- 期日が近い / 短時間でできるもの → `# 平日のタスク` または `# 週末のタスク`
- 曖昧なものは追記しない（リサーチノートの「次のアクション」に留める）

## Phase 4: アーカイブ（完了済みノートの整理）

`$ARGUMENTS` の内容に関係なく、毎回実行する。

### 4-1. 対象ノートの選択

1. `■メモ/🧭気になることリスト.md` の `### リサーチノート作成済み（チェック付きは対応完了）` を読む
2. その中で `- [x]` の行を完了対象として抽出する
3. 行内リンクが `[[■リサーチ/xxx]]` の場合は「未アーカイブ完了対象」として抽出する
4. 行内リンクが `[[■リサーチ/リサーチ完了/xxx]]` の場合は「アーカイブ済み」としてスキップする
5. 追加で `■リサーチ/*.md` を確認し、ノート内 `- [x] 対応完了` だがリスト未反映のものがあれば完了対象に含める
6. 該当がない場合は「アーカイブ対象はありません（Phase 4 は実行済み）」と表示して終了する

### 4-2. 完了状態の確認

- [x] 行に紐づくノートが `- [ ] 対応完了` の場合は `- [x] 対応完了` に更新する
- リンク先ファイルが存在しない場合は自動作成せず、警告として結果に出す（リンク必須ルール違反）
- 完了対象はユーザー確認なしでそのままアーカイブ処理へ進む

### 4-3. ノートをアーカイブ

1. `■リサーチ/リサーチ完了/` フォルダが存在しない場合は `Bash` で作成する
2. `Bash` で対象ノートを `■リサーチ/[名前].md` から `■リサーチ/リサーチ完了/[名前].md` に移動する
3. ノート内の `元: [[■メモ/🧭気になることリスト]]` を `元: [[リサーチ（アーカイブ済み）]]` に更新する（任意）
4. 気になることリスト側のリンクを `[[■リサーチ/リサーチ完了/[名前]]]` へ更新し、チェック状態は `- [x]` のまま維持する

### 4-4. リンクの更新

以下のファイルで旧パスへのリンクを新パスに書き換える。

- `■メモ/📆タスクリスト.md` — `[[■リサーチ/xxx]]` → `[[■リサーチ/リサーチ完了/xxx]]`
- `■メモ/🧭気になることリスト.md` — 同様に更新

## 注意事項

- 調査結果は事実と推測を区別して書く
- WebSearch の結果は出典URLを必ず記録する
- 既存のリサーチノートがある場合は上書きせず追記する
- 気になることリストの `作成済み` 項目はリンク必須。リンクなし状態を許容しない
- 気になることリストのリンク表記は `[[■リサーチ/xxx]]` または `[[■リサーチ/リサーチ完了/xxx]]` に統一する
- 気になることリストの項目自体は削除しない（セクション移動とリンク更新のみ）
- タスクリストへの追記は高確度のもののみ
- 結論は「確定」ではなく「現時点」の判断として記録し、前提が変わったら更新する
- 子エージェントの結果は必ず親エージェントが統合し、重複や矛盾を整理してから書き出す
- 絵文字は使わない
